# 质差定界派单工具设计文档 (统一接口与纯文本直出版)

## 概述

本文档定义了质差定界派单行为树中前三个节点的工具接口规范。
**核心架构原则：**
1. **接口高内聚**：智能体在每个节点只调用**一个**后端综合查询接口。后端接口负责执行底层所有必要的 SQL 查询，将基础业务信息与分析计算结果一并打包返回。
2. **重组装轻计算**：智能体拿到后端返回的全量数据字典后，**不进行任何计算**，直接按照规定的 Markdown 模板组装成纯文本。
3. **极简前端渲染**：各节点生成的 Markdown 文本依次追加存储到数据库的 `agent_output` 字段。前端直接读取该文本字段进行渲染，实现“所见即所得”。

## 节点①：质差识别检测 (detect_anomaly)

### 1.1 基本信息

| 属性 | 值 |
|------|-----|
| 工具名称 | detect_anomaly |
| 所属节点 | ① 质差识别检测 |
| 行为树返回值 | 无质差返回 Failure（流程结束），有质差返回 Success（进入节点②） |

### 1.2 后端接口返回数据字典 (API Response Data)
*后端接口需通过 SQL 关联查出以下所有基础配置与计算结果，一次性返回给智能体：*

| 字段分类 | 字段名 | 类型 | 说明 (后端SQL数据来源) | 示例值 |
|---|---|---|---|---|
| **基础信息** | `indicatorName` | string | 指标名称 (来自配置表) | "寻呼成功率" |
| | `thresholdType` | string | 阈值类型 (来自配置表) | "静态阈值" |
| | `thresholdRule` | string | 阈值规则描述 (来自配置表) | "93%" |
| | `timeGranularity` | string | 时间粒度 | "小时" |
| | `detectionPeriod` | string | 检测周期 | "6小时" |
| **分析明细** | `analysisList` | array | **时段明细列表** (后端计算事实表与配置表得出) | 见下说明 |
| ↳ *明细字段* | `stat_time` | string | 统计时段 | "2026-01-28 11:00" |
| ↳ *明细字段* | `metric_value` | string | 指标实际计算值 | "89.50%" |
| ↳ *明细字段* | `threshold_val` | string | 对比阈值 | "93%" |
| ↳ *明细字段* | `is_defect` | string | 是否质差 (后端判断后直接返回文字) | "是√" |
| **汇总结果** | `hasDefect` | string | 是否有质差时段 | "是√" |
| | `defectCount` | number | 质差时段总数 | 2 |
| | `defectPeriodsStr`| string | 质差时段列表(逗号分隔好的字符串) | "2026-01-28 11:00、2026-01-28 13:00" |

### 1.3 智能体纯文本拼装模板 (Agent Output Text)
*智能体获取上述数据字典后，遍历填入以下 Markdown 模板：*

```markdown
### ① 质差识别

— 输入数据 —
* 指标名称：{indicatorName}
* 阈值类型：{thresholdType}
* 阈值规则：{thresholdRule}
* 时间粒度：{timeGranularity}
* 检测周期：{detectionPeriod}

— 分析过程 —
| 时段 | 指标值 | 阈值 | 是否质差 |
| :--- | :--- | :--- | :--- |
| {stat_time_1} | {metric_value_1} | {threshold_val_1} | <font color="red">{is_defect_1}</font> |
| {stat_time_n} | {metric_value_n} | {threshold_val_n} | <font color="red">{is_defect_n}</font> |

— 输出结果 —
* 是否有质差时段：{hasDefect}
* 质差时段数：{defectCount}个
* 质差时段列表：{defectPeriodsStr}
```

## 节点③：智能定界定位分析 (execute_fault_localization)

### 3.1 基本信息

| 属性 | 值 |
|------|-----|
| 工具名称 | execute_fault_localization |
| 所属节点 | ③ 智能定界定位分析 |
| 行为树返回值 | 定界成功返回 Success（进入节点④），定界失败返回 Failure |

### 3.2 后端接口返回数据字典 (API Response Data)

| 字段分类 | 字段名 | 类型 | 说明 (后端SQL数据来源) | 示例值 |
|---|---|---|---|---|
| **预警详情** | `alertId` | string | 预警ID | "WARN-2026012804" |
| | `indicatorName` | string | 指标名称 | "寻呼成功率" |
| | `defectPeriodsStr`| string | 质差时段总览 | "2个 (11:00/13:00)" |
| | `customerInfo` | string | 客户名称与等级 | "广汽汽车集团（金牌）" |
| | `apnName` | string | APN名称 | "SCCAR.GD" |
| **流程控制** | `hasErrorCodeMapping`| string | 是否配置错误码映射表 | "是" / "否" |
| | `isErrorCodeInconclusive`| string | 错误码定界无明确结论 | "是" / "否" |
| **分析结论** | `errorCodeConclusion`| string | 阶段1结论 (后端查询错误码表并汇总文字) | "该指标未配置错误码定界规则..." |
| | `dimensionConclusion`| string | 阶段2结论 (后端执行聚类分析得出) | "通过聚类及横向比对分析，定界到无线网" |
| | `locationConclusion` | string | 阶段3结论 (定位出的具体对象名) | "定位疑似 小区ID 68272912 存在问题" |
| **输出结果** | `finalDomain` | string | 最终定界专业 | "无线网" |
| | `finalLocation` | string | 最终定位对象 | "疑似 小区ID 68272912 存在问题" |

### 3.3 智能体纯文本拼装模板 (Agent Output Text)

**动态逻辑说明：**
- 若 `hasErrorCodeMapping == '否'`：阶段1显示"未配置错误码映射表，跳过此阶段"
- 若 `hasErrorCodeMapping == '是'` 且 `isErrorCodeInconclusive == '是'`：阶段1显示结论并备注"继续执行聚类定界分析"
- 若 `hasErrorCodeMapping == '是'` 且 `isErrorCodeInconclusive == '否'`：阶段1显示结论并备注"直接定界到该专业"，阶段2前缀"*(基于阶段1结论，跳过聚类)*"

```markdown
### ③ 定界定位

— 输入数据 —
* 预警详情：
  * 预警ID：{alertId}
  * 指标：{indicatorName}
  * 质差时段：{defectPeriodsStr}
  * 客户：{customerInfo}
  * APN：{apnName}

— 分析过程 —
<font color="#4a90e2">**阶段1：错误码定界**</font>
⊙ **错误码定界** ➝ {根据hasErrorCodeMapping动态生成}
  ↳ *注：{根据isErrorCodeInconclusive动态生成}*

<font color="#4a90e2">**阶段2：聚类及关联分析**</font>
⊙ **维度定界** ➝ {stage2_prefix}{dimensionConclusion}

<font color="#4a90e2">**阶段3：定位对象**</font>
⊙ **具体定位** ➝ {locationConclusion}

— 输出结果 —
* 定界专业：{finalDomain}
* 定位：{finalLocation}
```

**动态文本生成规则：**

| 条件 | 阶段1文本 | 阶段2前缀 |
|------|----------|----------|
| `hasErrorCodeMapping == '否'` | "未配置错误码映射表，跳过此阶段。" | "" |
| `hasErrorCodeMapping == '是'` 且 `isErrorCodeInconclusive == '是'` | "{errorCodeConclusion}" + "\n  ↳ *注：错误码无明确结论，继续执行聚类定界分析。*" | "" |
| `hasErrorCodeMapping == '是'` 且 `isErrorCodeInconclusive == '否'` | "{errorCodeConclusion}" + "\n  ↳ *注：某专业维度占比 ≥ 40%，直接定界到该专业！*" | "*(基于阶段1结论，跳过聚类)* " |

## 附录：数据库存储执行标准

表结构维持原有的 `wo_order.js_order_check_agent_output` 不变，核心操作聚焦于 `agent_output` (TEXT) 字段。

**执行动作流：**
1. **触发查询**：智能体进入某一节点，向后端发送唯一标识（如 `order_id` 或 `alert_id`）调用该节点的专属数据接口。
2. **文本渲染**：智能体获取返回的数据字典 JSON，将变量注入对应的 Markdown 模板中。
3. **入库持久化**：
   * 节点①执行完毕后：执行 SQL `UPDATE ... SET agent_output = '拼接好的Markdown1' WHERE ...`
   * 节点②执行完毕后：执行 SQL `UPDATE ... SET agent_output = CONCAT(agent_output, '\n\n', '拼接好的Markdown2') WHERE ...`
   * 节点③执行完毕后：继续执行 SQL `UPDATE ... SET agent_output = CONCAT(agent_output, '\n\n', '拼接好的Markdown3') WHERE ...`
接口 + 纯模板渲染”的设计模式非常清晰，后端的归后端，智能体的归智能体。这套方案目前看来逻辑闭环非常完整，接下来您是否需要开始规划这三个接口具体的 Request/Response 数据结构定义（比如针对 Swagger 或 YApi 的对接文档）？
